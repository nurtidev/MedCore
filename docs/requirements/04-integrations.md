## Описание проблемы

Клиники сталкиваются с необходимостью **ручного переноса записей** из внешних приложений (например, iDoctor) в календарь клиники, что приводит к:
- Человеческим ошибкам
- Потере данных
- Низкой операционной эффективности

## Цель разработки

Обеспечить **автоматический обмен данными** между DCH и сторонними сервисами, включая бесшовную синхронизацию расписания клиники со сторонними сервисами лидогенерации.

## Функциональные требования

### Агрегаторы (iDoctor и др.)
- Автоматическое создание записей в календаре DCH при бронировании через iDoctor
- Двухсторонняя синхронизация статусов между системами
- Маппинг внешних ID врачей и услуг с внутренними ID системы

### Лаборатории (Олимп, Инвиво)
- Приём результатов анализов в формате PDF и структурированных данных (JSON/XML)
- Автоматическое прикрепление результатов анализов к электронной медицинской карте пациента

### Технические требования
- Система Webhooks для приёма запросов от агрегаторов и лабораторий
- Поддержка входящих вебхуков

## Ожидаемые результаты

| Метрика | Цель |
|---|---|
| Задержка появления записи | Не более 1 секунды |
| Двойное бронирование | Исключить полностью |
| Снижение ручного труда администраторов | 35–50% |
| Результаты анализов | Автоматическое прикрепление к карте |

## Технические требования из ТЗ

- **Backend:** Node.js *(мы реализуем на Go)*
- **База данных:** PostgreSQL
- **Интеграции:** iDoctor API, APIs партнёрских лабораторий (Олимп, Инвиво)

## Заметки для разработки

- Adapter pattern для каждого внешнего сервиса — единый интерфейс внутри
- Circuit breaker (sony/gobreaker) для всех внешних HTTP вызовов
- Retry с exponential backoff
- Входящие webhooks — верификация подписи от партнёров
- Dead letter queue в Kafka для failed integrations
- Кэширование справочных данных (маппинги ID) в Redis
